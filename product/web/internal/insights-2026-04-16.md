# LV2 Park — Project Retrospective
*April 16, 2026. Written after the build sprint.*

---

## What We Built

A fully automated static site that answers one question: is LV2 parking in effect near Wrigley today?

Stack: GitHub Pages (hosting), GitHub Actions (automation), Cloudflare Workers (form handling), Resend (email), Google Sheets (subscriber log), GA4 (analytics). Total cost: ~$1/month (domain only).

Shipped in roughly one sprint:
- Homepage with live status, 7-day schedule, map
- Live enforcement ticker (real-time LV2 ticket count from Chicago's public e-hearing portal)
- Email subscribe flow with welcome email + Sheets logging
- Monday digest with 7-day schedule
- GA4 traffic monitor with automated FlexOffers reapply alert
- Game recap auto-generator (one SEO page per game, morning after)
- Blog with 13 posts written and scheduled through July 9, all auto-publishing by date
- SYSTEM.md architecture doc

## Iterations and What Changed

### SpotHero button
Went through three versions before removing it entirely.

v1: Full-width 56px block button. Too dominant, visually loud.
v2: Compact inline pill (34px, inline-flex). Better. But still wrong.
v3: Removed. The question was never "is the button the right size?" The real question was "why is it here if there's no revenue?" Once that was asked, the answer was obvious. It goes back in when FlexOffers approves.

**Lesson:** Features without a revenue or UX reason are just noise. Ask "why is this here?" before asking "how do we style it?"

### Enforcement ticker
v1: Dark card with yellow badge (background #1A1A2E, LV2 CONFIRMED label). Looked dramatic but competed visually with the hero answer. Made a busy screen busier.
v2: Quiet inline text line. Better. But duplicate emoji appeared.
v3: Bordered inline text, no emoji. Clean, subordinate to the hero, still informative.

**Lesson:** Data displays should support the primary message, not compete with it. The hero answer ("NO" or "YES") is what the user came for. The ticker is supporting context.

### Email digest date filter
`week.json` contains the full Cubs season (180 days). The digest was showing 81 events. Fixed by filtering to `date >= today` and taking the first 7. The data was right; the consumer of the data was wrong.

**Lesson:** Always verify what the data file actually contains before writing the code that reads it. Don't assume "week" means 7 days.

### Google SA JSON secret
Pasting a multiline JSON file into `wrangler secret put` only stored the first line. Silently broke Sheets logging with no clear error. Fixed by piping from file: `cat credentials.json | wrangler secret put GOOGLE_SA_JSON`.

**Lesson:** Never paste multiline secrets into terminal prompts. Always pipe from file. This will bite you again on other projects.

### Worker confusion (old vs. new)
Two Cloudflare Worker files existed: `cloudflare-worker/worker.js` (old, Sheets only) and `worker/index.js` (new, full Resend integration). The Google Sheets logging had been added to the old file but not the new one. Led to a merge + rethink of which file was actually deployed.

**Lesson:** When there are two versions of the same thing, delete the old one immediately. Rename folders if needed. `cloudflare-worker/` should have been deleted when `worker/` was created.

### Bootstrap timeout
The ticket ID bootstrap (finding today's window center) runs 2,000 probes at 0.4s each = ~13 minutes. Ran it in background, stdout buffering made output disappear. Killed it manually after Phase 1 completed, wrote `ticker_state.json` by hand from the last known good value.

**Lesson:** Long-running bootstrap scripts need a checkpoint file that saves progress incrementally, not just a final result. If the script dies, you should be able to resume from the last checkpoint rather than restart.

### Blog publishing
Originally thought each post needed manual additions to the blog index on publish date. Turned out the index already does client-side date filtering. Posts in the array are invisible until their date arrives. The only step needed is adding the post to the array when it's created, which happens at write time.

**Lesson:** Read the existing code before building automation for something that's already automated.

## What Actually Worked Well

**The FOIA data as a competitive moat.** 9,434 LV2 tickets from 2018-2023, obtained before this site existed. No other site has this. It powers the heat map, data explorer, street rankings, and every game recap. This is the kind of asset that takes years to reproduce and can't be rushed. The lesson is: look for public data that requires effort to obtain (FOIA, scrapers, manual collection) because that effort is the moat.

**GitHub Actions as a free server.** The entire data pipeline runs on GitHub's free tier. Python scripts on a cron, committing JSON files to the repo, which GitHub Pages serves as a static site. No server to maintain, no database to manage, no bills. This architecture pattern works for any data-driven site where data changes slowly (daily or less).

**Client-side date filtering for scheduled content.** Embed all future posts in the JS array with publish dates. They're invisible until the date arrives. No CMS, no server, no cron job. Just a date comparison in the browser. Works for blog posts, promotional banners, seasonal messaging, anything time-gated.

**Resend + Cloudflare Workers for email.** Setup was under an hour. Free tier handles the current subscriber count easily. The subscribe flow (form → Worker → Resend + Sheets + Adam notification) all works without a backend server. Reusable pattern for any project that needs email capture.

**One question, one answer.** The homepage does one thing: tells you if LV2 is active today. That focus made every design decision easier. If a feature didn't directly answer that question, it was cut or subordinated.

## What We'd Do Differently

**Validate data file schemas before building consumers.** `week.json` containing 180 days instead of 7 was a surprise. Should have inspected the actual output of `fetch_data.py` before writing `send_digest.py`.

**Delete dead code immediately.** The old Cloudflare Worker should have been deleted the moment the new one was deployed. Dead code creates confusion that costs time later.

**Bootstrap should be resumable.** The ticker bootstrap is a one-time setup task but it's fragile. A 13-minute sequential probe with no checkpointing is not robust. For any future scraping/probing task of this length, write progress to a file every N iterations so it can be resumed.

**Start with a simpler live ticker.** The sequential ID scanning approach is clever but complex. A simpler first version could have been: count LV2 violations on the public ticket lookup by date with a known violation code. Less precise but much simpler to build and maintain. Could have shipped that faster and upgraded later.

**The "busy homepage" problem wasn't fully solved.** The user flagged that the homepage felt like a lot to look at. We made incremental improvements (removed SpotHero, simplified ticker) but didn't do a full hierarchy pass. The LV2 active state has: hero answer, LV2 pill, event list, day game note, ticker. That's five visual elements. Could probably be reduced to three.

## Money Tree Framework Alignment

### What's working

**PLG is native.** The site answers a question people are already googling. No sales, no outreach. Someone searches "is lv2 parking active today" and finds the answer. The subscribe flow is in-context. The email digest is useful, not promotional. This is the pattern.

**Distribution is clear.** SEO (game recaps compound over the season), email list (Monday digest keeps users coming back), direct (people bookmark it and check every game day). Three channels, all organic, all free.

**It runs itself.** After setup, zero manual work. Data updates 4x daily. Email goes out Monday. Recaps generate overnight. Blog posts publish by date. This is what "money tree" means operationally: it earns while you're doing other things.

**The stack is as simple as possible.** JSON files over a database. Static HTML over a framework. GitHub Actions over a server. Cloudflare Workers over a backend. Every choice pushed toward the minimum necessary complexity.

### Where it's not there yet

**No revenue yet.** FlexOffers rejected on traffic. SpotHero is paused. AdSense not applied. The site is built correctly but the monetization is blocked by traffic, which is blocked by Google indexing time. This is the risk of SEO-first businesses: the feedback loop is months, not days.

**Traffic is unproven.** We don't know yet if people actually search for LV2-related terms in volume. The FOIA data analysis and game recaps are bets on long-tail SEO. Could be right. Could take until mid-season to know.

**Email list is the real asset.** If 500 people subscribe and open the Monday digest, that's an audience for local sponsorships (parking garages, Wrigleyville bars, Cubs gear retailers). That's more reliable than affiliate revenue. The focus should be growing the list, not waiting for FlexOffers.

## The One Insight Worth Keeping

The live ticker isn't just a feature. It's the reason the site is unique. Anyone can pull the MLB schedule and tell you if there's a game today. Only this site tells you if enforcement is actually happening right now.

That's the insight that makes the product defensible. If you're building something where data is the product, ask: what data can you get that requires real effort, and what does that data tell people that they can't find anywhere else? That's where the moat is.
